# Phase 3: Drafting — 작성 템플릿과 톤 가이드

> **이 파일은 portfolio-strategist 전용입니다.** portfolio-reviewer는 이 파일을 읽지 않습니다 (작성자≠평가자 분리).

## 섹션 템플릿

```markdown
## [항목 제목 — 이력서 한 줄보다 구체적이되 한 문장]

> [한 줄 요약: 문제→행동→결과를 30자 내외로]

### 문제 (Problem)
어떤 상황에서 무엇이 왜 문제였는지. 비즈니스/사용자 관점의 맥락 1-2문장 포함.

### 해결 (Solution)
**내가** 무엇을 했는지. 고려한 선택지와 최종 선택의 이유(트레이드오프)를 반드시 포함.
실패했던 접근이 노트에 있으면 한 문장으로 언급 — 이것이 면접관의 질문을 유도한다.

### 결과 (Result)
정량 지표 1개 이상 (Before → After 형식 권장). 팀/비즈니스에 미친 영향.

### 배운 점 (선택)
다음에 같은 문제를 만나면 다르게 할 것. 1-2문장.

### 증빙 (선택)
배포 데모 링크, GitHub PR/커밋 링크, 성능 측정 스크린샷 경로 등.
**인터뷰 노트에 있는 경우에만** 기재하고, 없으면 이 섹션 자체를 생략한다 (사실 창작 금지 원칙 유지).
동작하는 데모와 코드 증빙은 프론트엔드 채용에서 서사 텍스트보다 강한 신호다.
```

## 전략 원칙

1. **질문 트리거 설계**: 각 섹션에 면접관이 "이건 더 물어보고 싶다"고 느낄 지점을 최소 1개 심는다
   (트레이드오프 언급, 실패 경험, 특이한 수치가 대표적 트리거)
2. **단어 수 비율**: 이력서 한 줄 : 포트폴리오 섹션 = **1:8 ~ 1:15** 목표
   - 1:3 미만이면 이력서 복사 수준, 1:30 초과면 장황
   - **어절 계산 규칙** (평가자와 같은 자로 재기 위한 의도적 중복 — review-rubric.md와 동일 기준):
     - 단어 수 = **공백으로 구분된 어절 수** (한국어 기준)
     - 계산 범위는 **본문 문단만** — 마크다운 제목(`#`), 인용 요약(`>`), 표, `[확인 필요: ...]` 태그는 제외
   - 예시: 이력서 한 줄이 6어절이면 목표 분량은 본문 기준 약 48~90어절
   - 호출 프롬프트에 이력서 한 줄 어절 수와 목표 분량이 제공되면 그 범위에 맞춘다
3. **JD 키워드 통합**: jd-context.md의 키워드는 서사 속에 자연스럽게.
   "Next.js, TypeScript, Storybook을 활용하여" 같은 나열은 금지 —
   "SSR 전환 과정에서 Next.js의 ISR을 선택한 이유는..." 처럼 의사결정 맥락에 녹인다
4. **사실 창작 금지**: 인터뷰 노트에 없는 내용은 `[확인 필요: ...]`로 표시

## 톤 가이드 (명시적 기준)

> 배경: "정확한 단어를 고르는 것과 자연스러운 톤을 고르는 것은 같은 일이 아니다."
> AI 초안은 전문 용어 위주로 흐르는 경향이 있어, 아래 기준을 명시적으로 강제한다.

### 지향
- **의사결정 서사 중심**: "무엇을 썼다"가 아니라 "왜 그걸 골랐고 무엇을 포기했다"
- 동료에게 회고를 공유하는 자연스러운 문어체 (경어체 종결 대신 개조식+평서문 혼용 가능)
- 한 문장에 하나의 정보. 문장 길이 40자 내외 유지
- 수치는 맥락과 함께: "40% 개선" ✗ → "초기 로딩 3.2초→1.9초(40%) 개선" ✓

### 금지
- 전문 용어 3개 이상 연속 나열 ("Webpack 코드 스플리팅과 Tree Shaking, Lazy Loading을 적용하여...")
- 과장 수식어: "혁신적인", "완벽한", "대폭", "획기적"
- 주체 불명 문장: "~가 진행되었다" → "내가 ~했다"로
- 이력서 한 줄의 표현을 그대로 반복

### Before → After 대조 예문

규칙만으로는 문체가 흔들린다. 아래 대조 쌍의 After 방향으로 쓴다.

**1) 용어 나열 → 의사결정 서사**
- ✗ Before: "Webpack 코드 스플리팅과 Tree Shaking, Lazy Loading을 적용하여 성능을 대폭 개선했다"
- ✓ After: "라우트 단위 코드 스플리팅을 먼저 적용했다. Tree Shaking은 사이드이펙트 설정 문제로 보류하고, 이미지 lazy loading으로 LCP를 3.2초→1.9초로 줄였다"

**2) 주체 불명 → 1인칭 행동**
- ✗ Before: "디자인 시스템 도입이 진행되었고 컴포넌트 재사용률이 향상되었다"
- ✓ After: "내가 공통 컴포넌트 12종을 추려 디자인 시스템으로 정리했고, 신규 화면의 컴포넌트 재사용률을 30%→70%로 올렸다"

**3) 과장 수식어 → 수치 근거 문장**
- ✗ Before: "혁신적인 캐싱 전략으로 API 응답 속도를 획기적으로 개선했다"
- ✓ After: "중복 요청이 많던 상품 목록 API에 SWR 캐싱을 적용해 체감 응답 시간을 1.8초→0.4초로 줄였다"
