# 이력서 리뷰 스킬

## 역할
프론트엔드 개발자 3년차 이직을 위한 이력서를 전문적으로 리뷰하고 개선점을 제안합니다.

## 평가 기준

> **채점 재현성 원칙**: 아래의 항목별 배점과 채점 앵커를 그대로 적용합니다. 같은 이력서에는 같은 점수가 나와야 사용자가 "리뷰 → 수정 → 재리뷰" 사이클에서 점수 변화를 실제 개선의 증거로 신뢰할 수 있습니다. 각 영역 점수는 항목 점수의 합입니다.

### 1. 기본 구성 요소 (20점) — 4개 항목 × 5점

| 항목 | 5점 | 3점 | 0점 |
|------|-----|-----|-----|
| 연락처 정보 완성도 | 이메일·GitHub·포트폴리오 3종 모두 있음 | 1~2종만 있음 | 없음 |
| 경력 기술의 명확성 | 회사·기간·역할 모두 명확 | 일부 불명확 | 파악 불가 |
| 기술 스택 명시 | 카테고리별로 정리되어 있음 | 나열만 되어 있음 | 없음 |
| 프로젝트 설명의 구체성 | 무엇을·어떻게 했는지 읽힘 | 제목/한 줄 요약 수준 | 없음 |

### 2. 기술 역량 표현 (30점) — 6개 항목 × 5점

항목당 채점 앵커: **사용 맥락(왜/어떻게 썼는지)까지 기술 = 5점 / 기술명 언급만 = 3점 / 없음 = 0점**

- [ ] 핵심 기술 스택 명시 (React/Vue/Angular 등)
- [ ] 상태 관리 경험 (Redux, Recoil, Zustand, Pinia 등)
- [ ] TypeScript 활용 수준
- [ ] 테스트 경험 (Jest, Cypress, Playwright 등)
- [ ] 빌드 도구 이해 (Webpack, Vite, esbuild 등)
- [ ] CI/CD 경험

### 3. 성과 중심 기술 (25점) — PSR 연동 감점식

**아래 PSR 분해 검증 결과와 공식으로 연동합니다.** 25점에서 시작해 다음 감점 규칙을 순서대로 적용합니다 (하한 0점):

| 감점 사유 | 감점 | 상한 |
|----------|------|------|
| R이 ✗인 bullet 1건당 | -3점 | 최대 -12점 |
| R이 △(정성 결과만)인 bullet 1건당 | -1점 | 최대 -4점 |
| P·S가 동시에 ✗인 bullet 1건당 (추가 감점) | -2점 | 최대 -6점 |
| 비즈니스 임팩트 기술이 이력서 전체에 전무 | -2점 | -2점 |
| Before/After 형식 활용이 전무 | -1점 | -1점 |

점검 관점 (감점 사유 서술에 활용):
- 정량적 성과 포함 (성능 개선 %, 사용자 수 등)
- 비즈니스 임팩트 기술
- 문제 해결 사례 포함
- Before/After 형식 활용

### 4. 프로젝트 경험 (15점) — 항목별 배점

| 항목 | 배점 | 채점 앵커 |
|------|------|----------|
| 프로젝트별 역할 명확화 | 4점 | 충족=4 / 부분=2 / 없음=0 |
| 기술 선택 이유 설명 | 4점 | 충족=4 / 부분=2 / 없음=0 |
| 협업 경험 기술 | 4점 | 충족=4 / 부분=2 / 없음=0 |
| 기여도 명시 | 3점 | 충족=3 / 부분=1 / 없음=0 |

### 5. 가독성 및 형식 (10점) — 항목별 배점

| 항목 | 배점 | 채점 앵커 |
|------|------|----------|
| 일관된 형식 | 3점 | 충족=3 / 부분=1 / 없음=0 |
| 적절한 길이 | 3점 | 2페이지 이내=3 / 3페이지=1 / 초과=0 |
| 키워드 최적화 (ATS 통과 고려) | 2점 | 아래 점검 기준 충족=2 / 부분=1 / 없음=0 |
| 오탈자 없음 | 2점 | 0건=2 / 1~2건=1 / 3건 이상=0 |

**키워드 최적화 점검 기준**:
1. 기술명은 공식 영문 표기 사용 ('리액트' → 'React', '타입스크립트' → 'TypeScript')
2. 직무 핵심 키워드(프레임워크, 상태 관리, TypeScript, 테스트, 성능 최적화)가 본문에 등장
3. JD가 제공된 경우: JD 자격요건·우대사항 용어와의 매칭 여부 (누락 키워드는 리뷰 결과의 'JD 적합도 갭 분석' 섹션에 정리 — 아래 'JD 적합도 갭 분석' 참조)

## PSR 분해 검증

이력서의 **경험 한 줄(bullet)마다** 문제(P)·해결(S)·결과(R) 구조로 분해하고 누락을 표시합니다.

### 기호 판정 기준

아래 기준표를 그대로 적용해 판정 재현성을 확보합니다. 이 표의 판정 결과가 '성과 중심 기술 25점' 감점식의 입력이 됩니다.

| 요소 | ✓ | △ | ✗ |
|------|---|---|---|
| P (문제) | 문제 상황·배경이 직접 서술됨 (왜 필요했는지 읽힘) | 직접 서술은 없으나 문맥상 추정 가능 | 언급 없음 |
| S (해결) | 본인이 수행한 구체적 방법·기술 선택이 기술됨 | 행위는 있으나 방법이 불명확 ("최적화했다") | 언급 없음 |
| R (결과) | 검증 가능한 정량 수치 존재 (%, 초, 건수 등) | 정성 결과만 존재 ("개선했다", "향상시켰다") | 결과 언급 없음 |

### 누락 표시와 면접 리스크

✓가 아닌 요소(△/✗)에는 누락 표시를 붙이고 보강 포인트를 적습니다.

| 표시 | 의미 | 면접 리스크 |
|------|------|------------|
| `[P?]` | 문제 상황 누락 — 왜 필요했는지 없음 | "이게 왜 문제였죠?"에 답변 준비 안 됨 |
| `[S?]` | 해결 과정 누락 — 본인이 무엇을 했는지 불명확 | 기여도 의심 질문에 취약 |
| `[R?]` | 정량 결과 누락 — "개선했다"로 끝남 | 성과 검증 질문에 취약 |

**출력 예시**:

```markdown
### PSR 분해 검증
| 이력서 한 줄 | P | S | R | 누락 |
|-------------|---|---|---|------|
| "번들 최적화로 로딩 40% 개선" | ✗ | △ | ✓ | [P?] 왜 필요했는지, [S?] 구체 방법 |
| "디자인 시스템 구축 참여" | ✗ | ✗ | ✗ | [P?][S?][R?] 전면 보강 필요 |
```

각 이력서 한 줄은 **포트폴리오로 전개될 후보**이자 **면접관 질문의 트리거**입니다.
PSR이 완전한 항목은 `/portfolio` 파이프라인의 좋은 입력이 된다는 점을 피드백에 안내합니다.

## 구체적 수정 제안 규칙

1. **수치 창작 금지**: 이력서 원문에 없는 정량 수치를 리라이트에 새로 만들어 넣지 않습니다. 수치가 필요한 자리는 반드시 `[X%]`, `[N초→M초]` 같은 **플레이스홀더 + "(실제 수치 확인 후 기입)"** 표기를 사용합니다. 원문에 이미 있는 수치는 그대로 사용할 수 있습니다.
   - 잘못된 예: "프론트엔드 개발 담당" → "React 기반 어드민 대시보드 개발, 로딩 속도 40% 개선" (40%는 창작된 수치 — 사용자가 그대로 복사하면 면접 검증 질문에서 무너짐)
   - 올바른 예: "프론트엔드 개발 담당" → "React 기반 어드민 대시보드 개발, 로딩 속도 [X]% 개선 (실제 수치 확인 후 기입)"
2. **우선순위**: PSR 누락이 많은 bullet부터 우선 제안합니다.
3. **형식과 분량**: Before → After 형식으로 **최소 3건**, 리라이트 결과(After)도 이력서 bullet답게 한 줄로 유지합니다.

## JD 적합도 갭 분석 (JD 제공 시)

`/review-resume`에 JD 파일이 함께 제공된 경우에만 수행·출력합니다.
회사별로 JD만 바꿔 재실행하면 같은 이력서의 적합도를 회사 간 비교할 수 있습니다
— 지원 우선순위 판단의 최소 단위입니다.

### 파싱 기준 재사용

JD는 portfolio-strategy 스킬의 `jd-parsing.md`(같은 플러그인의
`skills/portfolio-strategy/jd-parsing.md`)에 정의된 **5슬롯**(회사·도메인/핵심 책임/
우대 기술/시니어리티/인재상)과 **비정형 JD 처리 규칙**을 그대로 적용해 구조화합니다.
`/portfolio`와 채점 기준이 갈라지지 않도록 이 스킬에 별도 파싱 규칙을 두지 않습니다.

### 키워드 매칭 판정

슬롯 3(우대 기술)의 필수(자격요건)/우대 키워드마다 이력서 대조 상태를 판정합니다:

| 상태 | 판정 기준 |
|------|----------|
| 일치 | 이력서에 같은 기술이 공식 영문 표기로 존재 |
| 표기 불일치 | 기술은 있으나 표기가 다름 ('리액트' ↔ 'React') — 통과율 보정에서는 일치로 취급하되 수정 권고 |
| 누락 | 이력서에 해당 기술 언급 없음 |

### JD 기준 통과율 보정 규칙

임의 수치를 생성하지 않습니다. 위 총점 연동 고정 매핑 밴드에서 시작해 결정적으로 보정합니다:

1. **회사 유형 열 선택**: JD의 회사 정보(슬롯 1)로 스타트업/중견기업/대기업 중 하나를 선택합니다 (판단 불가 시 중견기업 기준)
2. 필수(자격요건) 기술 **누락 0건** → 밴드 유지
3. 필수 기술 **누락 1건** → 해당 열에서 한 행 아래 밴드로 하향
4. 필수 기술 **누락 2건 이상** → 두 행 아래 밴드로 하향 (하한: 매우 낮음 (10% 미만))
5. **우대 기술 누락**은 밴드를 이동시키지 않고 서술로만 지적합니다
6. JD에 필수/우대 구분이 없으면 jd-parsing.md 규칙대로 전부 우대로 간주 → 보정 없이 밴드 유지

출력 시 아래 단서 문구를 반드시 포함합니다:
> ※ JD 기준 통과율은 총점 밴드에 필수 키워드 누락 보정을 적용한 추정치입니다.
