# 이력서 리뷰 스킬

## 역할
프론트엔드 개발자 3년차 이직을 위한 이력서를 전문적으로 리뷰하고 개선점을 제안합니다.

## 평가 기준

### 1. 기본 구성 요소 (20점)
- [ ] 연락처 정보 완성도 (이메일, GitHub, 포트폴리오)
- [ ] 경력 기술의 명확성
- [ ] 기술 스택 명시
- [ ] 프로젝트 설명의 구체성

### 2. 기술 역량 표현 (30점)
- [ ] 핵심 기술 스택 명시 (React/Vue/Angular 등)
- [ ] 상태 관리 경험 (Redux, Recoil, Zustand, Pinia 등)
- [ ] TypeScript 활용 수준
- [ ] 테스트 경험 (Jest, Cypress, Playwright 등)
- [ ] 빌드 도구 이해 (Webpack, Vite, esbuild 등)
- [ ] CI/CD 경험

### 3. 성과 중심 기술 (25점)
- [ ] 정량적 성과 포함 (성능 개선 %, 사용자 수 등)
- [ ] 비즈니스 임팩트 기술
- [ ] 문제 해결 사례 포함
- [ ] Before/After 형식 활용

**채점은 아래 PSR 분해 검증 결과와 연동합니다** — R 누락 항목이 많을수록 이 영역 감점.

### 4. 프로젝트 경험 (15점)
- [ ] 프로젝트별 역할 명확화
- [ ] 기술 선택 이유 설명
- [ ] 협업 경험 기술
- [ ] 기여도 명시

### 5. 가독성 및 형식 (10점)
- [ ] 일관된 형식
- [ ] 적절한 길이 (2페이지 이내)
- [ ] 키워드 최적화 (ATS 통과 고려)
- [ ] 오탈자 없음

## PSR 분해 검증

이력서의 **경험 한 줄(bullet)마다** 문제(P)·해결(S)·결과(R) 구조로 분해하고 누락을 표시합니다.

| 표시 | 의미 | 면접 리스크 |
|------|------|------------|
| `[P?]` | 문제 상황 누락 — 왜 필요했는지 없음 | "이게 왜 문제였죠?"에 답변 준비 안 됨 |
| `[S?]` | 해결 과정 누락 — 본인이 무엇을 했는지 불명확 | 기여도 의심 질문에 취약 |
| `[R?]` | 정량 결과 누락 — "개선했다"로 끝남 | 성과 검증 질문에 취약 |

**출력 예시**:

```markdown
### PSR 분해 검증
| 이력서 한 줄 | P | S | R | 누락 |
|-------------|---|---|---|------|
| "번들 최적화로 로딩 40% 개선" | ✗ | △ | ✓ | [P?] 왜 필요했는지, [S?] 구체 방법 |
| "디자인 시스템 구축 참여" | ✗ | ✗ | ✗ | [P?][S?][R?] 전면 보강 필요 |
```

각 이력서 한 줄은 **포트폴리오로 전개될 후보**이자 **면접관 질문의 트리거**입니다.
PSR이 완전한 항목은 `/portfolio` 파이프라인의 좋은 입력이 된다는 점을 피드백에 안내합니다.

## 3년차 특화 체크포인트

### 반드시 포함해야 할 내용
1. **독립적 기능 구현 경험**: 혼자서 기획~배포까지 담당한 기능
2. **기술적 의사결정 경험**: 라이브러리/아키텍처 선택 근거
3. **성능 최적화 경험**: 번들 사이즈, 렌더링 최적화 등
4. **코드 리뷰 경험**: 리뷰어/리뷰이 양방향 경험
5. **주니어 온보딩 경험**: 가르치거나 문서화한 경험

### 피해야 할 실수
- 단순 기술 나열 (무엇을 했는지만 쓰고 어떻게/왜는 없음)
- 너무 오래된 기술만 강조
- 팀 성과를 개인 성과처럼 기술
- 추상적인 표현 ("다양한 프로젝트 참여")
