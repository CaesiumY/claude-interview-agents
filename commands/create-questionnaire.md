# 면접 질문지 생성 명령

이력서를 기반으로 맞춤형 면접 질문지를 생성합니다.

## 사용법
```
/create-questionnaire [이력서파일경로]
```

## 실행 절차

> **경로 규칙**: 입력 파일과 산출물 경로는 모두 **커맨드를 실행한 현재 작업 디렉토리 기준**입니다.

### 1단계: 이력서 파일 읽기
- `resumes/` 폴더 또는 지정된 경로에서 이력서 파일 읽기
- `portfolio/*/final.md`가 존재하면 함께 읽기 (질문 소스로 활용)
- `resumes/[이름]_review.md`(`/review-resume` 산출물)가 존재하면 함께 읽어, 그 **PSR 분해 검증 표**와
  **포트폴리오 전개 추천** 항목을 이력서 기반 질문(2단계 프로젝트 선정 · 3단계 에이전트 생성)의
  참고 자료로 활용합니다. 이 자료들은 3단계 호출 프롬프트의 '보조 소스'로 에이전트에 그대로 주입합니다.
  특히 [정량] 질문은 **R(결과)가 누락된 bullet**(`[R?]` 표시)을 우선 대상으로 삼습니다.
  파일이 없으면 기존 동작 그대로 진행합니다 (선택적 입력).

**예외 처리**:
- **파일 미발견**: 이력서 파일이 없거나 경로가 틀리면 `resumes/` 폴더의 파일 목록을 보여주고
  사용자에게 경로 확인을 요청한 뒤 **중단**합니다. 임의로 추측해 진행하지 않습니다.
- **[이름] 결정 규칙**: 산출물 파일명의 `[이름]`은 입력 이력서 파일의
  **베이스네임(확장자 제외)을 그대로** 사용합니다 (예: `resumes/kim.md` →
  `resumes/kim_questionnaire.md`). `/review-resume`의 `[이름]_review.md` /
  `[이름]_evaluation.md` / `[이름]_mock_[type]_[YYYY-MM-DD].md` 파일 패밀리와
  이름이 일치해야 `/evaluate`의 이력서·모의면접 평가 파일 탐색이 끊어지지 않습니다.
  이력서 본문에서 추출한 지원자 이름은 **문서 제목 표시용으로만** 사용합니다
  (예: 질문지 제목 `# ○○○님 맞춤 면접 질문지`) — 본문에 이름이 없으면 제목에도
  베이스네임을 사용합니다. 파일명에는 본문 추출 이름을 쓰지 않습니다.

### 2단계: 이력서 분석
1. **경력 분석**: 연차 확인 (기본 대상: 3년차 = 2~4년)
2. **기술 스택 추출**: 주요 기술, 프레임워크 식별
3. **프로젝트 분석**: **핵심 프로젝트 2~3개 선정**, 역할·성과 파악 — 이력서 기반 질문의 생성 단위
4. **특이사항 식별**: 이직 횟수, 공백기, 특별한 경험

**예외 처리**:
- **연차 불일치**: 분석 결과 2~4년차가 아니면 사용자에게 고지하고, 해당 연차에 맞게
  난이도 라벨 비율을 조정해 진행합니다. 조용히 3년차 기준으로 생성하지 않습니다.
  - 1~2년차: [스크리닝] 60% 이상 (기본기 정확성 위주)
  - 5년차 이상: [본선] 70% 이상 + 설계/아키텍처 질문 추가
- **프레임워크 판별**:
  - React와 Vue가 모두 있으면 **최근/주력 프로젝트에서 사용한 쪽**을 우선합니다
  - React/Vue 외 프레임워크(Svelte, Angular, 바닐라 JS 등)가 주력이면, 질문은행의
    개념 축(상태 관리·성능 최적화·컴포넌트 설계)을 해당 프레임워크 용어로 변형해 생성합니다

### 3단계: questionnaire-designer 서브 에이전트 호출

질문지 생성(정량 규칙 적용·공통/이력서 기반 질문 구성·연계형 4단계 전개·역질문 생성)은
**핀 고정된 `questionnaire-designer` 서브 에이전트에 위임**합니다. 구조 계약(연속 Q번호, 동일 질문 블록
구조, 태그 규칙, 답변 가이드 말미 분리)은 `/evaluate`의 3-에이전트가 파싱하는 형식이므로, 형식 안정성을
보장하려면 세션 모델이 아니라 모델이 고정된 에이전트가 생성해야 합니다(메인 루프는 직접 작성하지 않음).

호출 프롬프트에 1·2단계에서 수집·판별한 자료를 그대로 주입합니다:

```
Use the questionnaire-designer subagent to design this interview questionnaire:

## 생성 대상
- 이력서 전문: [1단계에서 읽은 이력서 전문]
- 질문 가이드: ${CLAUDE_PLUGIN_ROOT}/skills/interview-questionnaire/SKILL.md
- 표시용 지원자 이름: [이력서 본문에서 추출한 이름 — 없으면 파일 베이스네임]

## 이력서 분석 결과 (오케스트레이터 판별)
- 연차: [2단계 연차 판별 결과] + 난이도 라벨 비율: [SKILL 기본 스크리닝 40% : 본선 60% — 연차 예외 시 2단계 규칙 적용 결과]
- 주력 프레임워크: [2단계 프레임워크 판별 결과 + 근거]
- 핵심 프로젝트: [2단계에서 선정한 2~3개]
- 특이사항: [이직 횟수·공백기·특별한 경험 — 없으면 "없음"]

## 보조 소스 (있는 경우에만)
- 포트폴리오 final.md: [portfolio/*/final.md 전문 또는 "없음"]
- 리뷰 파일 PSR·추천: [resumes/[이름]_review.md의 PSR 분해 표·포트폴리오 전개 추천 항목 또는 "없음"]
- JD 컨텍스트: [지원 회사·JD가 알려진 경우 5슬롯 — 없으면 "없음"]

## 요청
문항 수 규칙(총 20~25문항 등 — SKILL.md가 아니라 questionnaire-designer 자체 생성 절차 2번이 기준)과
SKILL.md의 난이도 비율·4카테고리·브라우저/네트워크/보안 필수 포함·역질문 생성 규칙·출력 형식(구조 계약
6개 조항)을 그대로 적용해 질문지 마크다운 전문을 생성하세요.
반환 전 Q번호 연속성·문항 수 합계·난이도 비율·역질문 3개를 자체 검증하세요.
질문지 전문만 반환하고(H1 제목부터 시작) 앞뒤에 설명을 붙이지 마세요.
```

### 4단계: 반환물 구조 계약 검증

에이전트가 반환한 질문지 마크다운에서 구조 계약을 검사합니다(H1 제목
`# ...님 맞춤 면접 질문지`부터 끝까지를 파일 내용 후보로 사용):

1. **Q번호 연속성**: Q1부터 끊김·중복 없이 연속 부여되었는지
2. **답변란 존재**: 모든 질문 블록(Q번호)에 `**답변:**` 란이 있는지(역질문 섹션 제외)
3. **역질문 3개**: 역질문 섹션에 유형별(기술 문화·팀 상황·성장 연결) 3개가 있는지
4. (부가) 총 문항 20~25개, 브라우저/네트워크/보안 질문·프로젝트별 [정량] 질문 포함 여부

- 구조 계약 위반(Q번호 불연속, 답변란 누락, 역질문 3개 미달) 시 **위반 항목을 명시해 1회 재호출**합니다.
- 재호출도 실패하면 사용자에게 원인을 알리고 **중단**합니다 — 반쪽짜리 질문지를 저장하지 않습니다.

### 5단계: 결과 저장
- 질문지를 `resumes/[이름]_questionnaire.md` 파일로 저장 (UTF-8)
- 각 질문 블록 바로 아래에 답변란 포함, 예상 답변 포인트는 파일 말미 '답변 가이드'로 분리
  (스킬 가이드의 출력 형식 참조)
- **동일 경로에 파일이 이미 존재하면** 덮어쓸지 사용자에게 먼저 확인합니다.
  기존 파일을 보존하려면 `resumes/[이름]_questionnaire_[YYYY-MM-DD].md`로 저장을 제안합니다.

> **다음 단계 안내**: 질문지에 답변을 작성한 뒤 `/evaluate resumes/[이름]_questionnaire.md`로 평가를 받으세요.
> 임원/컬처핏 면접 대비는 `/mock-interview`를 사용하세요.

## 서브 에이전트 구성

이 커맨드는 플러그인의 서브 에이전트를 활용합니다:

| 에이전트 | 역할 | 산출 |
|----------|------|------|
| `questionnaire-designer` | 구조 계약을 지키는 질문지 설계 | 정량 규칙·4카테고리·역질문을 적용한 질문지 마크다운 전문 |

**위임 이유**: 커맨드(메인 루프)는 사용자의 세션 모델이 실행하므로, 생성 주체가 세션 모델이면
구조 계약(연속 Q번호·동일 질문 블록 구조·태그 규칙·답변 가이드 말미 분리) 준수가 모델에 좌우됩니다.
`/evaluate`의 3-에이전트가 파싱하는 형식이므로, 모델이 고정된 에이전트에 위임해 형식 안정성을 보장합니다.
인터랙티브 단계(1단계 파일 확인·2단계 연차/프레임워크 예외 고지·5단계 덮어쓰기 확인)는 사용자와의
상호작용이 필요하므로 메인 루프에 남깁니다.
